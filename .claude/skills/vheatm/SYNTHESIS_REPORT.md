## §1 Core Insight (with non-obviousness gate)

**The ULTIMATE VHEATM is not "v15 + recovered fragments." It is a re-layering of the corpus
along three orthogonal axes the original evolution never separated cleanly.**

- **Layer 1 (Core Loop)** — 5 Tikai Principles + 3 enforcement gates. Provides ~80% of audit
  value. Was the entire framework in v8-9.
- **Layer 2 (Specialist Lenses)** — Triggered, not defaulted. STRIDE/ATAM/FMEA/MAESTRO/etc.
  routed via ref 22. Was the v11-12 expansion.
- **Layer 3 (Meta-Defense)** — Framework auditing itself. Scales with AUDIT_TARGET_TIER.
  Was the v13-15 contribution.

**Non-obviousness test:** If you read v15 alone, the three layers are tangled — everything
appears equally mandatory. The user cannot tell what is core vs triggered vs tier-scaled.
v16's contribution is the explicit stratification, the per-tier reference reading order,
and the recovery of three trust anchors (v8/v10) the evolution dropped.

**Why not just "use v15 as-is":** v15 lost five things that materially reduce its usability
and trust calibration. These losses are not noted in v15's changelog because the framework
never audited its own SKILL.md for institutional memory regression. See §3 below.

## §3 Opposing Synthesis — the strongest case AGAINST v16

> "The ULTIMATE VHEATM is v9 or v10. The post-v10 additions are 80% diminishing returns.
> Tikai's 5 Principles + ENTERPRISE awareness + 14 references already solve real audit
> failures. Everything after v10 is the framework optimizing for its own self-audit at the
> expense of being usable. [E.IJ] requires a separate LLM session that 90% of users don't
> have. [M.AT] needs 5 cycles before its threshold becomes empirical. [M.EP] runs 3 of 12
> rotating questions per cycle — sophisticated but irrelevant for the typical single-pass
> user. The right ultimate is: v10 baseline + AI-S5 + Independent Generation Protocol +
> [M.AT] + Heuristic Acknowledgment. DELETE [E.IJ], [M.EP], [M.BA], [KB.CMA],
> AUDIT_TARGET_TIER, [G.AB] vocabulary-expansion mode. The framework is 3.4× larger than v8
> and the marginal audit-quality improvement is unproven."

**This opposing case is genuinely strong.** v16 explicitly addresses each point:

1. **"Most additions are diminishing returns"** — Probably true *for typical use*. v16 mitigates
   by making AUDIT_TARGET_TIER orthogonal to Mode and providing per-tier reading order. Tier 1
   user reads ~5 refs and skips [E.IJ]/[M.EP]/[KB.CMA] entirely. The framework scales DOWN
   honestly, not just up.

2. **"[E.IJ] requires unavailable infrastructure"** — Partially true. v16 documents the
   constraint explicitly in the applicability note. Tier 1 skips it. Tier 2 marks it
   "recommended." Tier 3 makes it mandatory (these users typically DO have API access).

3. **"[M.AT] needs 5 cycles"** — True. PRE-CALIBRATION phase uses 60% industry-benchmark
   threshold (T3 evidence — Devo SOC report). Single-cycle users get conservative threshold,
   not no threshold. This is graceful degradation, not theater.

4. **"v8/v9 were enough"** — Not for AI-integrated systems. AI-S1–S5 (refs 11, 28) close a
   genuine attack surface not present in 2023. [G.AD] addresses adversarial code comments
   that v8-10 don't anticipate. This is qualitatively new threat space, not refinement.

**Where the opposing case wins:** Cognitive load is real damage. v16 caps it via:
- Minimum 3 entry fields (CONTEXT_MODE, STAKEHOLDER, GOAL); others default.
- Mode Router with explicit page-size and time-budget estimates (recovered from v8-10).
- Per-tier reading order showing Tier 1 needs only ~5 refs.

## §6 Attacks on Core Insight (4 mandatory)

### Attack 0 — Pre-mortem carry

**Failure mode:** "The user adopts the new structure, sees 'Layer 3 = scales with tier,' picks
Tier 1, and skips Layer 3 entirely. Then their AI auditor drifts (no [G.AD], no IGP) and they
don't know why findings are inconsistent."

**v16 mitigation:** [G.AD] Auditor Defense and Independent Generation Protocol are documented
as **always required for AI-executed audits, regardless of tier**. Only the empirical-
calibration parts ([M.AT] EMPIRICAL phase, [M.EP] full 12Q rotation, [KB.CMA] periodic) scale
with tier. Tier 1 still gets [G.AD] inline (1 question in FAST mode) and IGP for any AP lens.

### Attack 1 — Internal

**Failure mode:** "Three-axis structure is just relabeling. The 31 refs still exist, the 21
gates still exist. Cognitive load is the same."

**v16 mitigation:** Per-tier reading order is concrete (Tier 1 = ~5 refs, Tier 2 = ~15, Tier 3
= all 31). Minimum-3-fields entry protocol is concrete. Mode Router with page-size estimates
is concrete. Three of the v15 surface losses were exactly this kind of "everything appears
equally important" issue — v16 directly addresses scannability.

### Attack 2 — Anti-target Stakeholder

**Stakeholder:** A small-team CTO running their first audit on a SaaS startup app.

**Their objection:** "I'm at a 6-person startup. I want technical signal in 30 minutes. I see
'Attestation Artifact with 6-month expiration' and think 'SOC 2 vendor.' I see 'AUDIT_TARGET_
TIER 3 requires [E.IJ] via separate LLM context' and think 'I don't have that infrastructure.'
The whole thing feels like enterprise compliance theater I have to fight through to get to a
real audit."

**v16 addresses by:**
- AUDIT_TARGET_TIER defaults to 2 (Production), explicitly NOT 3 (Critical). Tier 3 is for
  critical infrastructure, not "any production app."
- Attestation is **mandatory output for Tier 2-3 only**; Tier 1 (MVP) outputs the Heuristic
  Acknowledgment note without expiration date.
- Mode FAST + Tier 2 = ~1 page output in 5-15 minutes. The CTO's actual use case.
- [E.IJ] applicability note explicitly addresses the "I don't have the infrastructure" case.

### Attack 3 — Falsifiability

**What evidence would refute v16's core claim?**

The core claim is: **three-layer re-stratification + recovered v8/v10 principles + tier-
specific reading order = strictly better than v15 alone for the typical user.**

**Falsifying observations:**
1. If users running v16-Tier-1 produce systematically lower-quality audits than users running
   v15 with same Mode, the layering hurt them by inviting under-investment in Layer 2-3.
   → Test: 10 audits each, blind-compared by external reviewer.
2. If the per-tier reading order causes users to skip refs they actually needed (e.g., Tier 1
   user skipping ref 28 → missing AI-S5 in an AI-integrated codebase), the tiering is wrong.
   → Mitigation: AI-S5 is in Layer 1 (always required for AI executors).
3. If [E.IJ] applicability documentation causes users to skip [E.IJ] when they could have
   used the API-in-artifacts workaround, honesty backfired.
   → Test: track [E.IJ] adoption rates pre/post v16.
4. If the recovered Anti-Patterns section (25 items) is too long and users ignore it,
   v15's 9 inline items were correct.
   → Test: which anti-pattern violations are reported in audit cycles?

v16 explicitly invites these falsification tests via Debt-INV7-001 and the [M.AT] tracker.
A framework that survives skepticism is stronger than one that avoids it (v14 6th Audit OS Truth).

v16-ULTIMATE is:
- **Complete** — every mechanism from v8-v15 traced and preserved or explicitly deprecated.
- **Cleaner** — five regressions from v12+ recovered; five stale labels in v15 fixed.
- **More usable** — three-layer architecture + per-tier reading order + minimum-3-fields

Stranger Review — the 5 PRISM v5.0 adversarial mechanisms.

**Residual confidence: 78% (hot) / 72% (cold).** Below the 80% threshold flagged by PRISM
v5.0 Confidence Decay Test — explicit acknowledgment per Humility Index. The 22-28% residual
risk is dominated by:
- [E.IJ] effectiveness not independently validated (~10%)

- Recovered v8-v10 principles may not survive contact with users who only know v15 (~4%)

These are tracked debts, not undocumented risks. A framework that surfaces its own residuals
honestly is the only kind that can improve.
